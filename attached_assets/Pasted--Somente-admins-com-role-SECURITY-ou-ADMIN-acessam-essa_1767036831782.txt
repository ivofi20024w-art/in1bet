🔥 PROMPT (KYC INTERNO PROFISSIONAL)

Contexto do projeto:
Plataforma de casino online brasileira com PIX, wallet, ledger, bônus e dashboard admin já funcional.

Objetivo deste update:
Implementar um SISTEMA DE KYC INTERNO COMPLETO, com DEPARTAMENTO DE SEGURANÇA DEDICADO dentro do /admin, nível operador profissional.

Princípios obrigatórios:
	•	KYC NÃO pode alterar saldo
	•	KYC NÃO pode creditar/descreditar dinheiro
	•	KYC apenas controla PERMISSÃO DE SAQUE e RISCO
	•	Tudo deve ser auditável
	•	Estrutura preparada para futura integração com APIs externas

⸻

1️⃣ MODELO DE DADOS (KYC)

Criar entidade KycVerification:
	•	userId
	•	cpf
	•	documentType (RG | CNH)
	•	documentFrontUrl
	•	documentBackUrl
	•	selfieUrl
	•	status (PENDING | APPROVED | REJECTED)
	•	rejectionReason (nullable)
	•	reviewedBy (adminId)
	•	reviewedAt
	•	createdAt

Garantir:
	•	Apenas 1 KYC ativo por usuário
	•	Histórico preservado (não sobrescrever)

⸻

2️⃣ FLUXO DO USUÁRIO

Criar páginas:
	•	/perfil/verificacao

Funcionalidades:
	•	Upload guiado de documentos (frente, verso, selfie)
	•	Validações básicas de arquivo (tipo, tamanho)
	•	Status visível em tempo real
	•	Mensagens claras:
	•	“Verificação pendente”
	•	“Verificado”
	•	“Rejeitado – enviar novamente”

Regras:
	•	Usuário pode depositar e apostar sem KYC
	•	Usuário NÃO pode sacar sem KYC aprovado

⸻

3️⃣ BLOQUEIO DE SAQUE (INTEGRAÇÃO)

Integrar com sistema de saque:
	•	Se KYC ≠ APPROVED → saque bloqueado
	•	Mensagem clara: “Verificação de identidade pendente”
	•	Admin vê exatamente o motivo do bloqueio

⸻

4️⃣ DEPARTAMENTO DE SEGURANÇA (ADMIN)

Criar nova aba /admin/seguranca

Subsecções:

🔍 KYC Pendentes
	•	Lista paginada
	•	Filtros por data, CPF, usuário
	•	Visualização completa dos documentos
	•	Acesso ao perfil financeiro do usuário

Ações:
	•	Aprovar KYC
	•	Rejeitar KYC (campo obrigatório de motivo)

⸻

✅ KYC Aprovados
	•	Histórico de verificações
	•	Admin responsável
	•	Data de aprovação

⸻

❌ KYC Rejeitados
	•	Motivo
	•	Tentativas
	•	Possível flag antifraude

⸻

5️⃣ AUDITORIA & LOGS DE SEGURANÇA

Criar logs específicos:
	•	Ação (approve / reject)
	•	Admin responsável
	•	Usuário afetado
	•	Data/hora
	•	Motivo

Logs imutáveis.
